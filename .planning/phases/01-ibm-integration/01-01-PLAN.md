    Implement backend/services/granite_service.py with EXACTLY these components (per locked decisions in CONTEXT.md):

    1. GraniteError exception class — wraps all SDK exceptions, decouples callers from IBM internals

    2. Re-export GraniteResponse from models.forge (so callers can import both from granite_service)

    3. Module-level singleton pattern:
       - _credentials: Credentials | None = None
       - _model: ModelInference | None = None
       - _get_model() function that lazily initializes both using settings from config.py:
         - Credentials(url=settings.watsonx_url, api_key=settings.watsonx_api_key)
         - ModelInference(model_id=settings.watsonx_model_id, credentials=_credentials, project_id=settings.watsonx_project_id, validate=False)
         - validate=False per Research Pitfall 5 — avoids double API call at startup

    4. async generate_text() coroutine:
       - Parameters: prompt (str), call_name (str, default "unknown"), max_tokens (int | None), temperature (float | None)
       - Build params dict using GenTextParamsMetaNames (MAX_NEW_TOKENS from max_tokens or settings.max_tokens_craft, TEMPERATURE if provided)
       - Use asyncio.get_running_loop().run_in_executor(None, lambda: model.generate_text(prompt=prompt, params=params))
       - Wrap in asyncio.wait_for with timeout=30.0
       - Measure wall time with time.perf_counter() for latency_ms
       - Catch asyncio.TimeoutError -> raise GraniteError(f"Granite call timed out ({call_name}): exceeded 30s")
       - Catch Exception -> raise GraniteError(f"Granite call failed ({call_name}): {exc}") from exc
       - Return GraniteResponse(text=text, latency_ms=round(latency_ms, 1))

    5. MUST NOT contain any os.getenv() calls — all config via `from backend.config import settings`
    6. Use asyncio.get_running_loop() (not get_event_loop) per Research state-of-the-art recommendation
